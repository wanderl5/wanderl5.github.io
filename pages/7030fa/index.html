Git基本介绍

# What is Git?

  • Git 是SCM（Software Configuration Management，软件配置管理）
  • 功能上同CVS，SVN，BitKeeper（商用）；
  • 设计原理大量借鉴BitKeeper；

As you learn Git, try to clear your mind of the things you may know about other VCSs, such as CVS, Subversion or Perforce

Git起步 (opens new window)

# Git独特之处

  • Branch and Merge（分支和合并处理）
  • Small and Fast（体积小,速度快）
  • Distributed （分布式,近乎所有操作都是本地执行,不受限于网络）
  • Data Assurance （数据可靠）
  • Free And Open Source（开源免费）

Git直接存储变动后文件，并每次提交都记录为1个快照(快照由tree和blob文件组成)，而非记录差异

# 状态变化

  • 未跟踪（untracked）：新文件 ，未进行add或commit
  • 已暂存（staged）：git add .后提交到暂存区, 文件状态是 staged, 显示为new file; (反操作： git restore --staged 1.txt )
  • 已修改（modified）： 修改已add或commit的文件；(反操作： git restore 1.txt ， 文件会回到之前)
  • 已提交（committed）： 执行 git commit完成提交后, 文件状态变成已提交, 完成暂存区内容提交本地仓库；(反操作： git rest --hard HEAD~1 ， 文件会回到之前)

注意： commit提交的内容是之前已add到暂存区的文件，而非本地最新文件

# 回退和撤销操作

# revert

  • git restore --staged <file> 把文件从暂存区移动到工作区，即文件不被追踪;
  • git restore <file>文件仍在暂存区且会撤销文件修改的内容，错误加入缓冲区的文件撤销提交(配合.gitignore修改后无法生效问题)

# reset

  • 撤销add到缓存区
# 单文件撤销 (缓存区的修改,针对已add文件)
git reset HEAD file.txt

# 撤销上次的所有add操作
git reset HEAD

# diff版本比较

  • git diff 什么参数都不加，默认比较工作区暂存区的差异
  • git diff --cached [<path>...]比较暂存区与最新本地版本库（本地库中最近一次commit的内容）
  • git diff HEAD [<path>...]比较工作区与最新本地版本库。如果HEAD指向的是master分支，那么HEAD还可以换成master
  • git diff commitid [<path>...]比较工作区与指定commit-id的差异
  • git diff --cached commit1 [<path>...]比较暂存区与指定commit-id的差异
  • git diff commit1 commit2 比较两个commit-id之间的差异
  • git diff commitid1 commitid2 --stat查看两个提交版本id修改了那些文件.
  • git diff 版本号码1 版本号码2 src 比较两个版本的src 文件夹的差异
# 查看两个id间差别的文件
git diff 78a15b1 4f7d9d2 --stat

 include/testLib/common.cpp                  |  18 +-
 include/testLib/common.h                    |  26 +++
 xprice/TestCode/swap/TestSwapWorkAction.cpp | 325 ++++++++++++++++------------

# 查看具体差异
git diff 78a15b1   include/testLib/common.cpp

# diff制作补丁

git diff还可以制作补丁文件，在其他机器上对应目录下使用git apply patch将补丁打上即可,便于在不push的情况下直接同步代码

  • git diff > patch patch的命名是随意的，不加其他参数时作用是：将我们本仓库工作区的修改拷贝一份到其他机器上使用

  • git diff --cached > patch是将暂存区与版本库的差异做成补丁

  • git diff --HEAD > patch是将工作区与版本库的差异做成补丁

  • git diff filename > patch将单个文件做成一个单独的补丁

  • git apply patch应用补丁。

注意： 应用补丁之前我们可以先检验一下补丁能否应用，git apply --check patch 如果没有任何输出，那么表示可以顺利接受这个补丁。 使用git apply --reject patch将能打的补丁先打上，有冲突的会生成.rej文件，此时可以找到这些文件进行手动打补丁。
